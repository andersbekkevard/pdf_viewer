# ADR 0007 — Native browser find via cached shadow text

**Status**: Accepted
**Date**: 2026-04-24

## Context

The rolling render window from ADR 0002 hides non-buffered page content with:

```css
.pf > .pc { display: none !important; }
```

That is the correct paint/performance contract for large pdf2htmlEX outputs,
but Chromium native find does not index text inside `display:none`. Result:
`Cmd-F` only searches the current render window unless the user toggles
render-all, which is exactly the heavyweight path we try to avoid for
several-hundred-page textbooks.

The overlay already has `/` search for visible-page work. This decision is
only about native browser `Cmd-F`: make it index the full document while
keeping the visible pdf2htmlEX render window narrow.

## Decision

Add a cache-side per-page plain-text artifact:

```text
~/.cache/pdf_viewer/<hash>/text.json
```

Schema v1:

```json
{
  "version": 1,
  "source": "pdf2htmlEX-html",
  "pages": [
    { "page": 1, "text": "..." }
  ]
}
```

Generate it from the converted pdf2htmlEX HTML, not from the source PDF.
Fresh conversion, serial indexing, parallel indexing, and
`upgrade-cache.sh --mode=text` all produce or backfill it.

Page numbers are derived from the hex `#pf...` page ID. Do not trust
`data-page-no`: in the pinned pdf2htmlEX output it is also hex, and values
like `data-page-no="140"` mean decimal page 320.

At runtime, `assets/overlay.js` fetches `/<hash>/text.json` and appends one
clipped text node under each page wrapper:

```html
<div id="pf2a" class="pf" data-page-no="2a">
  <div class="pc">...</div>
  <div class="pdf2html-find-shadow" aria-hidden="true">...</div>
</div>
```

The shadow node is visually hidden via clipping, not `display:none`, so it
remains visible to Chromium's find indexer while the real page content is not
rendered. Once a page is rendered, CSS hides that page's shadow node so native
find sees one source per page: real `.pc` text when available, shadow text
otherwise.

The fetch uses `cache: "no-store"` because `text.json` can be regenerated by
cache upgrades independently of the injected HTML. Serving stale page-number
maps silently breaks full-document find.

When native find lands in a shadow node, the overlay detects the resulting
`selectionchange`, force-renders that page, maps the selected shadow-text
occurrence back into the real pdf2htmlEX `.t` text nodes, and replaces the
selection with the real range. The shadow layer remains the index; the active
selection should become real text.

## Rationale

**Cache-side extraction keeps the read path light.** The expensive walk over
pdf2htmlEX text nodes happens once per cache entry or explicit upgrade, not
on every document open.

**Extracting from HTML avoids source-PDF coupling.** Remote cache entries may
have expired signed URLs, and local sources may have moved. The converted HTML
is the canonical readable artifact and already contains the selectable text
that copy/yank workflows use.

**Per-page placement anchors browser find to the PDF page.** A single global
transcript would be simpler, but native find navigation would scroll to that
transcript rather than the corresponding page. Keeping shadow text under each
`.pf` gives the browser a page-local layout target.

**The daemon stays read-only.** No route extracts text or invokes tooling.
This preserves ADR 0004: conversion and cache mutation stay in scripts.

## Consequences

### Accepted downsides

- **Extra cache bytes.** `text.json` duplicates plain text already present in
  the HTML. This is small relative to pdf2htmlEX's image/font output.
- **Extra DOM text nodes.** The overlay mounts one shadow node per text-bearing
  page. This is far cheaper than rendering every `.pc`, but not free.
- **The selection bridge is best-effort.** It maps by selected text plus
  occurrence index within the page. This is reliable for normal word/phrase
  searches, but pathological repeated text or browser find normalization we do
  not replicate could select the wrong occurrence.
- **Browser find internals are still browser-owned.** The overlay can replace
  the page Selection after a shadow hit, but Chromium still owns the find box,
  match count, and next/previous state.

### Wins

- `Cmd-F` can search the full cached document without render-all.
- Existing `/` search remains scoped to visible pages.
- Existing cache entries can be upgraded without Docker or source PDFs.
